## 一、硬件架构

```
你的 F103C8T6 开发板
  → AHT20 温湿度采集
  → 旋钮/按键输入
  → CAN 发送 (ID=0x12, 1Hz)
        │ CAN 500kbps
        ▼
你的 F407ZGT6 (金星开发板 KF56-12)
  ┌─ Bootloader (0x08000000, 32KB) ──────┐
  │   → 检查 ota_flag                     │
  │   → =1: 通过 CH340C 串口收固件 (Ymodem)│
  │   → =0: boot_check_stack2jump_app()   │
  └────────────────────────────────────────┘
  ┌─ APP (0x08008000, ~480KB) ────────────┐
  │   → FreeRTOS 任务管理                  │
  │   → LVGL 仪表盘 (精简版 GUI)           │
  │   → CAN 接收 (F103 数据)              │
  │   → SPI Flash (W25Q128) 图片          │
  │   → 检测 OTA 命令 → 设 ota_flag → 复位  │
  └────────────────────────────────────────┘
```

### 角色分配

| 角色 | 芯片 | 职责 |
|------|------|------|
| CAN Sender | **F103C8T6**（你手上的板） | AHT20、旋钮、按键 → CAN 总线 |
| Bootloader | **F407ZGT6** | CAN + OTA + 串口固件接收 |
| Main APP | **F407ZGT6** | LVGL 显示 + CAN 接收 + 业务逻辑 |

### 例程 vs 本级差异

| 项目 | uint3code 原版 | 本级 |
|------|---------------|------|
| Bootloader 芯片 | F103ZE (512KB) | F407ZGT6 **(1MB)** |
| APP 芯片 | F103ZE (512KB) | F407ZGT6 **(1MB)** |
| USB Tool 芯片 | F103C8 (额外的) | **F407 自身 CH340C 替代** |
| CAN Sender 芯片 | F103C8 | F103C8 **(复用)**
| 固件传入方式 | CAN + ISO-TP（经 USB Tool） | **CH340C 串口 + Ymodem** |
| 内存资源 | Flash 512KB / SRAM 64KB | Flash **1024KB** / SRAM **192KB** |
| 显示 | TFT LCD | NT35510 800×480 **已有** |
| SPI Flash | 无 | W25Q128 **16MB 已有** |

## 二、F407 1MB Flash 分区

```
地址范围                    大小     内容
0x08000000 - 0x08007FFF    32KB     Bootloader（向量表 + 初始化 + OTA 逻辑）
0x08008000 - 0x080DFFFF    864KB    APP 主程序
0x080E0000 - 0x080FFFFF    128KB    参数扇区（Sector 11, ota_flag+Magic+Checksum）
                                    ⚠ 擦除以 128KB sector 为单位!
```

**关键约束：**
- 参数扇区位于 Flash 最顶部 Sector 11，独立于 Bootloader 和 APP
- 参数扇区的 APP 分区外，APP 擦写不会误擦参数
- APP 的中断向量偏移 `VECT_TAB_OFFSET = 0x8000`

## 三、实施方案（5 Phase + 后续）

### Phase 1：Bootloader + APP 分区跳转（骨架） ✅ 已完成

**目标**：验证双分区启动机制，Bootloader 能跳转到 APP 运行 LVGL。

**步骤：**

1. **调整链接脚本**（STM32F407ZGTx_FLASH.ld）
   - APP 工程：`FLASH (rx)` 起始地址改为 `0x08008000`，长度 `0x78000`
   - Bootloader 工程：`FLASH (rx)` 起始地址 `0x08000000`，长度 `0x8000`

2. **移植 `boot_manager.c`**
   - `boot_check_stack2jump_app(addr)` — MSP 设置 + 中断关闭 + 函数指针跳转
   - 栈顶合法性检查: `(*(addr) & 0x2FFE0000) == 0x20000000`
   - 支持 AC5 (ARMCC) 和 AC6 (ARMClang)

3. **APP 侧修改**
   - `SystemClock_Config` 后调用 `HAL_SetSysTick()` 修正 SysTick 中断
   - `main.c` 中调 `SCB->VTOR = 0x08008000` 或 HAL 自动处理

4. **⚠ Bootloader 跳转前必做**：
   - 关闭所有外设中断（`__set_PRIMASK(1)`）
   - **关闭 SysTick 中断**（否则跳转后 SysTick 中断向量指向 Bootloader 区，产生 HardFault）
   - 反初始化 HAL（`HAL_DeInit()` 清理 RCC 和 SysTick 配置）

**验证方法：**
- 分别编译 Bootloader 和 APP
- 先烧录 Bootloader（`0x08000000`），再烧录 APP（`0x08008000`）
- 复位后观察串口输出 `[BOOT]--->jump app`，然后 APP 启动 LVGL

**涉及新增文件：** `boot_manager.c`, `boot_manager.h`
**涉及修改文件：** `STM32F407ZGTx_FLASH.ld`（APP 版 + Bootloader 版）

---

### Phase 2：参数扇区管理

**目标**：实现 Flash 参数区的读写、Magic 校验、ota_flag 管理。

**步骤：**

1. **创建 `inter_flash_cfg.c/h`**
   ```c
   typedef struct {
       uint8_t magic[4];       // 0xAABBCCDD
       uint32_t ota_bin_version;
       uint8_t ota_flag;       // 0=正常, 1=更新
       uint8_t checksum;       // 累加和校验
       uint8_t format[2];      // 占位对齐
   } inter_flash_cfg_param_typeDef;
   ```

2. **创建 `inter_flashif.c/h`（F407 适配版）**
   - F1 系列 vs F4 系列 Flash 操作差异：
     | 操作 | F1 (例程) | F4 (本级) |
     |------|-----------|-----------|
     | 擦除页大小 | 1KB (page) | **16KB (sector)** |
     | 编程单位 | `FLASH_TYPEPROGRAM_WORD` | `FLASH_TYPEPROGRAM_WORD`（兼容） |
     | 擦除函数 | `HAL_FLASHEx_Erase` type=`FLASH_TYPEERASE_PAGES` | `FLASH_TYPEERASE_SECTORS` |
   - 实现：`inter_flashif_read_page`, `inter_flashif_write_page`, `inter_flashif_erase_page`
   - 注意：F407 的 16KB sector 最小擦除单位，参数区只用一个 sector

3. **Bootloader 中集成**
   - 启动时 `inter_flash_cfg_init()` 校验 Magic + Checksum
   - `inter_flash_cfg_get_app_update_flag()` 判断是否进 OTA

**验证方法：**
- Bootloader 启动打印 `flash cfg size:12`、Magic 校验结果
- 手动写 ota_flag=1，复位后 Bootloader 应进入等待更新状态

**涉及新增文件：** `inter_flash_cfg.c/h`, `inter_flashif.c/h`

---

### Phase 3：串口 Ymodem 固件接收 + Flash 写入（核心 OTA）

**目标**：通过 CH340C 串口接收 PC 发送的固件，写入 APP 分区。

**步骤：**

1. **移植 Ymodem 协议**
   - `ymodem.c/h` — 直接复制，纯 C 无硬件依赖
   - `ymodem_porting.c` — 实现串口收发（复用 USART1 已有驱动）
   - 支持 128 字节 (SOH) 和 1024 字节 (STX) 数据包
   - CRC16 校验

2. **实现 Flash 写入流程**
   ```c
   for (每个 Ymodem 包) {
       inter_flashif_erase_page(addr + offset);  // 按 sector 擦除
       inter_flashif_write_page(addr + offset, data, len/4);
   }
   ```

3. **Bootloader 主循环**
   ```
   检测 ota_flag == 1
     → printf("wait ota...")
     → Ymodem_Receive()
     → 写入 Flash
     → 完成后清除 ota_flag
     → boot_check_stack2jump_app(APP_ADDR)
   ```

4. **PC 端操作流程**
   1. APP 工程编译出 `.bin`
   2. 打开串口工具（SecureCRT/Tera Term）
   3. 选 Ymodem 协议发送 `.bin` 文件
   4. Bootloader 接收完成后自动跳转

**验证方法：**
- 通过串口发一个测试 bin（如 LED 闪烁程序）
- 观察串口输出 `Programming Completed Successfully`
- 复位后应运行新程序

**涉及新增文件：** `ymodem.c/h`, `ymodem_porting.c`

## 七、F407 关键硬件配置参数（施工前必读）

> 🔴 **本章节记录了历次踩坑验证后的最终正确配置，所有参数均经实测确认。不按此配置会导致通信失败、显示异常甚至硬件损伤。**

### 7.1 CAN 总线（F407 → F103，500kbps）

**引脚配置（与 F103 不同！）：**
```
F407: CAN1_RX(PA11) = GPIO_MODE_AF_PP + GPIO_AF9_CAN1    ← 必须！
      CAN1_TX(PA12) = GPIO_MODE_AF_PP + GPIO_AF9_CAN1
      两个引脚一次配置，Pull=NOPULL, Speed=VERY_HIGH
      
F103: CAN_RX(PA11)  = GPIO_MODE_INPUT                      ← F103 方式
      CAN_TX(PA12)  = GPIO_MODE_AF_PP
```

> 🚨 **最大坑**：F407 的 CAN_RX 不能用 F103 的 INPUT 模式！否则 ESR 全零但收不到任何数据（耗时 2 小时排查）。详见 PROGRESS.md #13。

**时序计算（500kbps）：**
| 参数 | F407 | F103 |
|------|------|------|
| APB1 时钟 | 42MHz | 36MHz |
| Prescaler | 6 | 12 |
| tq | 0.143μs | 0.333μs |
| BS1 | 11 TQ | 4 TQ |
| BS2 | 2 TQ | 1 TQ |
| SyncSeg | 1 TQ | 1 TQ |
| Bit Time | (1+11+2)×0.143=2.0μs | (1+4+1)×0.333=2.0μs |
| **最终波特率** | **500kbps** ✅ | **500kbps** ✅ |

**CAN 过滤器：全接收模式**
```c
// F407 + F103 相同
FilterMode = CAN_FILTERMODE_IDMASK;
FilterScale = CAN_FILTERSCALE_32BIT;
FilterIdHigh = 0x0000;   FilterMaskIdHigh = 0x0000;
FilterIdLow  = 0x0000;   FilterMaskIdLow  = 0x0000;
// Mask=0 → 所有 ID 通过，软件层区分
```

**CAN 帧格式（F103 → F407，ID=0x12）：**
| Byte0 | Byte1 | Byte2 | Byte3 | Byte4 | Byte5 |
|-------|-------|-------|-------|-------|-------|
| 温度整数+40 | 温度小数×10 | 湿度整数 | 湿度小数×10 | 旋钮值 | 按键事件 |

**帧 ID 定义（当前项目）：**
| ID | 方向 | 用途 |
|----|------|------|
| 0x12 | F103→F407 | 传感器数据帧（温湿度+旋钮+按键） |

详见 [docs/CAN.md](file:///E:/ST/STM32/MY_workspace/Projects_F407/MY_OTA_GUI/docs/CAN.md)

**CubeMX 再生防护 ⚠️：**
- 每次 CubeMX 重新生成后，检查 `HAL_CAN_MspInit` 中 CAN1_RX 是否为 **PA11**（不是 PB8！）
- 检查是否有残留的 `HAL_GPIO_Init(GPIOB, ...)` 配置行
- 检查 `guider_ui`、`guiTaskHandle`、`touchTaskHandle`、`canRxTaskHandle` 定义是否丢失

### 7.2 SPI Flash W25Q128（SPI1，PB3/PB4/PB5/PB14）

**本板实测芯片：Winbond W25Q128JV，JEDEC ID = `0xEF6018`（`0x9F` 命令）**

| 参数 | 配置 |
|------|------|
| SCK( PB3) / MISO(PB4) / MOSI(PB5) | AF5(SPI1), 推挽, **PULLUP**, 速度 **MEDIUM**(25MHz) |
| CS(PB14) | 推挽输出, 软件控制 |
| NRF_CS(PG7) | **必须输出 HIGH**（防 NRF24L01 干扰 SPI 总线） |
| SPI 模式 | **Mode 3** (CPOL=1, CPHA=1) |
| 初始化速率 | **fPCLK/256 ≈ 328KHz** → 确认通信后提速 |
| 全速速率 | **fPCLK/2 ≈ 42MHz**（APB2=84MHz），已验证稳定 |
| 写入单位 | 页编程 256 字节 |
| 擦除单位 | 扇区 4KB |

> 🚨 **GPIO 速度陷阱**：SPI GPIO 用 `VERY_HIGH`(100MHz) 会引起过冲/串扰导致通信出错！`MEDIUM`(25MHz) 在 42MHz SPI 下完全稳定。详见 [SPI_Flash_W25Q128芯片信息](file:///E:/ST/STM32/MY_workspace/知识库/工程实用信息/F407信息/SPI_Flash_W25Q128芯片信息.md) 踩坑记录第 4 条。

**关键 SPI Flash 命令：**
| 命令 | 字节数 | 说明 |
|------|--------|------|
| `0x06` | 1 | 写使能 (WREN) |
| `0x03` | 1+3+N | 读数据 |
| `0x02` | 1+3+N | 页编程 (≤256字节) |
| `0x20` | 1+3 | 扇区擦除 4KB(~45ms) |
| `0x9F` | 1+3 | 读 JEDEC ID → 0xEF6018 |

**JTAG 冲突**：PB3(PB3/SWO) 和 PB4(JNTRST) 默认被 JTAG 占用，CubeMX SYS Debug 必须选 **"Serial Wire"** 释放这两个引脚。

**SPI 长事务保护**：长 SPI 操作（擦除/多页写入）前后用 `__disable_irq()` / `__enable_irq()` 保护，防止 TIM2 中断破坏 CS 时序。详见 [SPI_Flash_W25Q128芯片信息](file:///E:/ST/STM32/MY_workspace/知识库/工程实用信息/F407信息/SPI_Flash_W25Q128芯片信息.md) 踩坑记录第 5 条。

**S: 路径格式**：
```
"S:<offset_hex>:<size_hex>"  例: "S:000A0000:0001C200"
```

### 7.3 LCD NT35510（FSMC NE4）

| 参数 | 配置 |
|------|------|
| 基址 | `0x6C000000`（NE4=PG12） |
| RS 选择 | PF12(FSMC_A6)，struct 偏移技巧区分命令/数据 |
| 数据宽度 | 16-bit parallel |
| 分辨率 | 480×800（竖屏原生） |
| 命令格式 | **16-bit 命令字**（不是 MIPI DCS 8-bit！） |
| 背光 | PB15, 推挽输出, 高电平点亮 |
| FSMC 引脚 | 所有 FSMC 引脚 `PULLUP | SPEED_FREQ_HIGH` |

**命令/数据地址计算**：
```c
#define TFTLCD_BASE  ((uint32_t)(0x6C000000 | 0x0000007E))
typedef struct {
    uint16_t LCD_CMD;   // *(base+0) → A6=0 → RS=0
    uint16_t LCD_DATA;  // *(base+2) → A6=1 → RS=1
} TFTLCD_TypeDef;
// 0x7E 确保 CMD 地址 bit7=0, DATA 地址(+2) bit7=1
```

**初始化顺序（严格！）：**
```
1. 上电等待 50ms（内部 POR）
2. 发命令 0x0100 软件复位（SWRESET）← 必须！POR不可靠！
3. 等待 120ms
4. 退出睡眠 0x11，等待 120ms
5. 像素格式 0x3A → 0x55 (16-bit RGB565)
6. 访问控制 0x36 → 0x00 (默认RGB顺序)
7. 显示开启 0x29
```

> 🚨 **花屏根因**：缺少 SWRESET(0x0100) 导致寄存器不确定状态，像素写入随机错色。加上 SWRESET + 120ms 等待后完全正常。

**FSMC 覆盖 KEY 引脚**：CubeMX FSMC 会将 PE0-PE4 全部配为 AF_PP。`MX_FSMC_Init()` **之后**必须重配 PE2-PE4 为 `GPIO_MODE_INPUT | GPIO_PULLUP`（KEY0-KEY2）。

详见 [F407_LCD_NT35510_GT5688芯片信息](file:///E:/ST/STM32/MY_workspace/知识库/工程实用信息/F407信息/F407_LCD_NT35510_GT5688芯片信息.md)

### 7.4 触摸 GT911（软件 I2C）

| 参数 | 配置 |
|------|------|
| SCL(PB0) | 推挽输出 |
| SDA(PF11) | 动态切换输入/输出 |
| RST(PC13) | 推挽输出（位带操作 `PCout(13)`） |
| INT(PB1) | **INPUT + NOPULL**（非 OUTPUT！） |
| I2C 地址 | 7-bit `0x14` (写 0x28 / 读 0x29) |
| 寄存器地址 | **16 位** |
| 芯片 PID | **"911"**（不是 "5668"） |
| 坐标变换(横屏) | `X = 800 - chip_y`, `Y = chip_x` |

> 🚨 PB1 是 GT911 的 INT 输出脚，MCU 必须配为 INPUT 模式。如果配成 OUTPUT_PP 会与芯片电平冲突导致 I2C 通信失败。

**触摸读取协议：**
```
1. 读 0x814E → 1 byte (status)
2. points = byte & 0x0F
3. if (byte & 0x80): 写 0x00 到 0x814E 清状态 ← points=0 也要清！
4. if (points > 0 && points < 6): 读每点坐标
```

> 🚨 **只清 points>0 的状态不清 points=0 的** → 芯片永久卡在"空事件"状态，再也不报新触摸。

**独立 FreeRTOS task 配置：**
```c
// 触控 task 必须独立于 LVGL task
osThreadAttr_t touchTask_attributes = {
    .name = "touchTask",
    .stack_size = 128 * 4,    // 512 bytes
    .priority = osPriorityAboveNormal,  // 高于 GUI task
};
```

### 7.5 外部 SRAM IS62WV51216（FSMC NE3）

| 参数 | 配置 |
|------|------|
| 基址 | `0x68000000`（NE3=PG10） |
| 容量 | 1MB (512K×16-bit) |
| NBL0(PE0)/NBL1(PE1) | **必须配置**（否则 16 位访问不可用） |
| 所有 FSMC 引脚 | GPIO_MODE_AF_PP(AF12), PULLUP, SPEED_FREQ_HIGH |

**地址线交叉（原理图确认）：**
```
SRAM_A10 ↔ FSMC_A17 (PD12)
SRAM_A17 ↔ FSMC_A10 (PG0)
→ 连续线性 buffer 上限 256KB
→ 块扫描 (≤64KB + DSB) 贯穿全 1MB 可靠
→ 不支持 LVGL DIRECT 全帧缓冲模式
```

**FSMC 时序：**
```
AddressSetupTime = 0 (1 HCLK)
AddressHoldTime = 0
DataSetupTime   = 0x18 (24 HCLK, ~143ns)
AccessMode      = Mode A
```

### 7.6 供电架构（重要！）

```
DC_IN 7-12V → POWER开关 → VCC5(5V) → 1117-3.3 → VCC3.3(3.3V)
                    │
                    ├── TJA1040 CAN 收发器 (5V)
                    ├── LCD 背光 (5V)
                    ├── 电机/舵机 = 共电源！← 启动电流拉低 VCC5 → LCD花屏
                    └── (NRF24L01, ESP8266 等模块)

VCC3.3(3.3V) ──┬── STM32F407ZGT6
               ├── IS62WV51216 SRAM (1MB)
               ├── W25Q128 SPI Flash (16MB)
               ├── NT35510 LCD
               ├── GT911 触摸
               └── LAN8720A Ethernet
```

> 🚨 **电机/舵机与 LCD 共用 5V 是本板的结构性限制**。电机启动电流 ~500mA 瞬间拉低 VCC5 → FSMC 总线噪声 → LCD 花屏。三个方案均因此失败。**如后续需驱动电机，必须使用独立电源**。

## 十一、注意事项与陷阱（非硬件相关）

### Bootloader 跳转
- 跳转前 **关闭所有中断**（`__set_PRIMASK(1)`）
- **关闭 SysTick 中断**并停止 SysTick 计数器，否则跳转后 SysTick 异常
- `HAL_DeInit()` 清理 RCC 和 SysTick
- 外设不会自动复位——Bootloader 中重新初始化所有外设

### 串口冲突
- Bootloader 和 APP 共用 USART1（CH340C）
- Bootloader OTA 接收完成后、跳转前需关闭串口中断

### Bootloader 尺寸
- 目标 < 32KB（Flash 分区 32KB）
- Bootloader 不含：LVGL、FreeRTOS、SPI Flash 驱动、Touch、GUI
- 仅含：HAL 初始化 + CAN + USART + Flash 操作 + Ymodem

### 工作原则
- 🔴 **遇到不懂的不要靠猜，有问题就及时和我沟通** — 本计划书中的配置参数是实测验证过的，但工程场景千变万化，不确定时先问
- 每次 Phase 开始前，先读此计划书的相关章节
- 只修改 `USER CODE BEGIN` ~ `USER CODE END` 之间的代码
- 不重新生成 CubeMX 硬件初始化代码除非必要
